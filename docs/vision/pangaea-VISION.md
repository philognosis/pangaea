# Pangaea — Vision, Northstar & Engineering Principles
**This document is the first thing Claude Code reads before touching any code.**
**Every implementation decision should be traceable back to something in here.**

---

## What Pangaea is

Pangaea is the operating system of the organization.

Not a portal. Not a search engine. Not an integration platform. An OS.

Just as a computer's OS provides applications with identity, storage, display,
and inter-process communication so that no application has to solve those
problems itself — Pangaea provides every internal capability team with intent
routing, identity delegation, trust, rendering, and push delivery, so that no
team has to build a frontend, wire SSO, pass a security review, or solve
accessibility ever again.

The geological name is deliberate. Pangaea was one landmass before continental
drift broke it into fragments. Internal enterprise software drifted the same
way — one capability per app, each on its own island, each with its own
interaction grammar, its own URL to remember, its own auth to wire. Pangaea
reassembles the landmass. The continents keep their terrain. The ocean
between them disappears.

---

## The northstar — one sentence

> **Any person in the organization can discover, access, and complete any work
> they are permitted to do — from one place, by expressing a human need,
> without knowing what systems exist.**

Every line of code either moves toward this sentence or it doesn't.

## The end goal — what done looks like

An employee opens Pangaea on their phone. No URL to remember. No app to
install (it's a PWA — it installs from one tap). What they see is a home
screen that knows who they are: pending work surfaced from every connected
system, quick actions relevant to their role, a search/voice bar at the top.

They say: *"I want to recognize Maya for the platform work she did."*

They don't know which system handles recognition. They don't know if it's
called "recognition," "rewards," "kudos," or "kudos.internal.company.com".
They just said what they want.

Pangaea's concept graph traverses the org's semantic map, finds that
`acme.rewards` claims the concept `recognition → peer_acknowledgement`,
routes to it, resolves "Maya" to Maya Chen (E-29183, Team Atlas) via the
entity spine, and streams back a guided form — scoped exactly to what the
caller is allowed to do.

The employee fills in two fields and taps Submit. A short-lived token is
minted for exactly this action. The rewards system's API is called. A
provenance receipt is sealed. A push notification lands on Maya's lock screen.
The whole thing took 45 seconds. Nobody typed a URL. Nobody logged in again.
Nobody read documentation.

That's done.

The org's 40 internal systems are all federated. Every team's integration cost
was one YAML file and one afternoon. Security reviewed the platform once;
every team inherits that review. Approval cycle times dropped from days to
minutes because work arrives at people instead of waiting for people to
arrive at work. And the failed-intent analytics tell every team exactly
what their users need that doesn't exist yet — the best product roadmap
they've ever had, generated by the people they serve.

## How Pangaea is different from what exists

**It is not enterprise search.**
Search finds the page. Pangaea finishes the task. These are different
categories. Search is pull-only and link-returning. Pangaea is push-capable,
action-executing, and receipt-producing.

**It is not an integration platform (MuleSoft, Boomi, etc.).**
Integration platforms wire systems to systems. Pangaea wires humans to systems,
through intent. The protocols are different: one is API-to-API; the other is
natural language to the right API, with identity, confirmation, and audit.

**It is not a UI component library or design system.**
A design system still requires teams to build frontends. Pangaea eliminates
the frontend entirely for teams that federate. The A2UI catalog is not a
library teams import — it's the renderer the platform uses on their behalf.

**It is not an AI chatbot.**
A chatbot is a conversational interface to a single system. Pangaea's
conversational surface is one of four entry points (voice, chat, push,
home screen) to every system in the org. The model's job is routing and
confirmation, not answering questions.

**The right comparison is an OS.**
iOS gave developers identity (Sign in with Apple), push (APNs), rendering
(UIKit/SwiftUI), and payments (StoreKit) so that no app had to build those
from scratch. Every app on iOS is better because of it. Pangaea does the
same thing for internal capabilities. The platform's power compounds with
every team that federates.

## The core architectural ideas — never compromise these

### 1. System calls, not tool calls

When a user expresses a need, Pangaea resolves it through the concept graph
to the right *system*, then to the right *operation* within that system —
exactly like an OS kernel dispatching a syscall through the right device driver.

The user never names a system. The platform discovers it. This is the
distinction between a tool-calling LLM and an operating system.

Each "syscall":
- Carries a short-lived, scope-limited, per-user OBO token
- Is recorded in the provenance ledger
- Returns typed, schema-validated data (not free text)
- Is invisible to the user unless they ask for the provenance footer

### 2. Teams own the verbs. Pangaea owns distribution and trust.

A team's capability manifest is their driver registration. It says:
"Here is what I can do, here are my entities, here is how to call me,
here is my risk tier for each action."

Once registered, Pangaea handles everything distribution-related:
discovering the capability, routing to it, rendering its data, pushing
its notifications, brokering its identity, auditing its actions.

The team never loses ownership. Their API is still the enforcement point.
Their data never passes through Pangaea at rest — only in transit.
Their domain agent, if they have one, is a first-class participant,
not a backend to someone else's platform.

### 3. The concept graph is the PATH

In Unix, the PATH is what lets you type `git` without knowing where git lives.
The concept graph is Pangaea's PATH. It's a semantic map of concepts →
capabilities. When a user expresses a need with no system named, the concept
graph is what finds the right system.

It must be:
- Automatically populated from manifests (no manual curation, no committee)
- Queryable by semantic similarity (embeddings, not keyword)
- Live-updated when new capabilities federate (new system auto-discoverable
  within seconds of `xp publish`)
- The source of truth for "what can Pangaea do?"

### 4. Model proposes. Deterministic surface confirms. Human commits.

For any action above `read` risk, there is exactly one path to execution:

```
model resolves intent
  → deterministic confirmation card rendered from resolved entity data
  → human commits with explicit tap
  → OBO token minted for exactly this action
  → domain API called
  → receipt sealed
```

The model never commits an action directly. The confirmation card is
rendered from data the platform fetched and validated — it is never
generated by the model. This makes consequential actions safe even when
intent resolution is probabilistic.

### 5. Content plane and control plane never mix

Data flowing from domain systems is typed payload bound to A2UI component
schemas. It is never interpreted as instructions by any part of the platform.
A field in a staffing record cannot influence what the orchestrator does next.
This is the structural defense against prompt injection — it's enforced by
architecture, not by policy.

### 6. Compiled paths are the performance contract

The LLM is in the cold path only. Once an intent route stabilizes (after N
uses with the same slot pattern), it is compiled into a deterministic Redis
entry. The second time a user says the same thing, the model is not called.
The route resolves in <50ms.

This means the system gets faster as it is used, and heavy users (who use
the same intents repeatedly) have the best experience — not the worst.

### 7. The entity spine is mandatory

Every composite view, every cross-system query, every disambiguation decision
anchors to the org's canonical entity spine — typically the directory/IdP
(Keycloak in dev, Okta/Azure AD in production). Every domain system references
entities from this spine; they never own the canonical identity.

"John Doe" resolves to exactly one record (or presents a disambiguation card
with employee IDs, teams, and reporting lines) before any action is taken.
This is a hard invariant. No syscall is dispatched with an unresolved entity.

## The federated model — why teams will join

Teams join because the value exchange is asymmetric in their favor:

**They give:** one YAML manifest file, one afternoon of integration work,
a commitment to publish an OpenAPI doc (which they should have anyway).

**They receive:**
- Voice, chat, push, and PWA surfaces for free
- SSO and delegated identity for free
- Security review amortized across the platform (their review shrinks from
  months to days because the rendering and auth layer is pre-certified)
- WCAG 2.2 AA accessibility across all their surfaces, for free
- Signed audit trail for every action, better than any app they've built
- Intent telemetry: the first time they see real product analytics on their
  internal capability — what people actually ask for, including what doesn't
  exist yet

The pitch to a team is: "Stop paying the UI tax. Publish a manifest. Get
everything else for free. Keep owning your data and your rules."

The political unlock: teams are not demoted to backends. They federate a
first-class domain agent. The platform never touches their domain logic.
"Federate the experience, not the ownership."

## What Pangaea is not responsible for

These are explicit non-goals. Do not build them. Do not let scope creep
pull the platform toward them:

- **Domain logic.** Pangaea never decides who should be evaluated, which
  candidate matches a role, or whether an extension should be approved.
  Those decisions live in domain systems and domain agents.

- **Storing domain data at rest.** Pangaea caches for performance (Redis,
  prefetch). It never stores domain records persistently. The domain system
  is always the system of record.

- **Governance of domain schemas.** Pangaea validates that manifests are
  structurally correct and internally consistent. It does not impose how
  a team models their domain.

- **Cross-domain transactions (sagas) before Phase 3.** This is a Phase 3
  feature, and only when telemetry proves the demand. Do not wire saga
  coordination before you have evidence it's needed.

- **A formal OWL/RDF ontology with a standing committee.** The concept graph
  is lightweight JSON-LD populated automatically from manifests. If teams
  have to attend meetings to maintain it, the architecture has failed.

## The four entry points — and their hierarchy

Every interaction with Pangaea comes through one of four surfaces. They are
listed in order of value delivered per user-second:

1. **Push notifications with action cards.** Work arrives at the user. They
   tap Approve on their lock screen. The task is done before they open an
   app. This is the highest-leverage surface and the first thing built.

2. **Home surface.** The ambient state of the user's work world. Pending
   items, role-relevant capabilities, suggested actions. This is what users
   see when they open Pangaea with no specific intent.

3. **Voice and chat.** The user expresses a need in natural language. The
   platform routes it, resolves entities, dispatches the task, renders the
   result. The conversational surface is for intentional, initiated workflows.

4. **PWA shell.** The browsable, visual surface. Available for users who
   want to navigate rather than express. Every federated capability is
   reachable here. Layout is pinned by default (spatial memory is a feature).

## How to read the test gates

Every phase ends with a test gate. The test gate is not a checkbox — it is
the demo. When you run the phase gate test with `-v` on a terminal in front
of stakeholders, each passing assertion is a visible beat of the story:

```
PASSED  push notification arrived in browser          ← they can see it
PASSED  user tapped Approve                           ← on the actual UI
PASSED  staffing stub received approved request       ← real API call
PASSED  OBO token had correct scopes, no extras       ← trust proven
PASSED  receipt written to ledger                     ← audit proven
PASSED  chain hash verified                           ← tamper-evident proven
```

Build test gates before building features. The test gate is the spec.

## Naming conventions — use these everywhere

| Concept | Name to use | Never use |
|---|---|---|
| The platform | Pangaea | "the system", "the platform" alone |
| A team's integration file | capability manifest | "config", "spec", "schema" |
| Semantic map of concepts | concept graph | "ontology", "taxonomy", "knowledge graph" |
| A team's intelligent participant | domain agent | "microservice", "backend" |
| Resolving intent to a system | routing | "searching", "querying" |
| Calling a domain system | syscall | "tool call", "API call" (in user-facing copy) |
| Short-lived delegated token | OBO token | "service token", "API key" |
| Signed action record | provenance receipt | "audit log", "event log" |
| UI built from component refs | A2UI surface | "generated UI", "dynamic UI" |
| The org's identity anchor | entity spine | "master data", "golden record" |

## The invariants — things that must always be true

These are not guidelines. If any of these is ever violated, stop and fix it
before continuing:

1. **No standing credentials.** The platform holds no persistent tokens to any
   domain system. Every token is minted per-request, scoped to one operation,
   TTL ≤ 600 seconds.

2. **No code in A2UI surfaces.** A2UI renders from component IDs and typed props
   validated against Zod schemas. No JSX, no eval, no dangerouslySetInnerHTML,
   no script tags, no markdown-with-embedded-html ever reaches the renderer.

3. **No unresolved entities before dispatch.** Every slot in an intent must
   resolve to a canonical entity ID before the orchestrator dispatches a
   syscall. Ambiguous entities produce a disambiguation card. Never guess.

4. **No model-generated confirmation cards.** Confirmation cards for
   consequential actions are built from data fetched and validated by the
   platform. The model never writes the text that appears in a confirmation card.

5. **Every consequential action has a receipt.** If an action with risk tier
   `consequential` or `reversible` completes without a receipt row in the
   ledger, it is a bug. The ledger write is not optional.

6. **Manifests are the source of truth.** If a capability's behavior is not
   declared in its manifest, the platform does not support it. The manifest
   is not documentation — it is the contract.

7. **The concept graph is populated automatically.** No human ever manually
   edits the concept graph. It is derived entirely from published manifests.
   If a team wants their capability to be discovered differently, they update
   their manifest.

## For Claude Code: how to use this document

Before writing any code, ask: *"Which invariant does this touch?
Which northstar principle does this serve?"*

When you are uncertain between two implementations, choose the one that
better preserves the invariants and moves more directly toward the northstar.

When a task in BUILDPLAN.md seems to conflict with something in this
document, this document wins. Raise the conflict as a comment before
proceeding.

When naming things — variables, functions, services, API paths, database
columns — use the naming conventions table above. Consistency across the
codebase is how a future engineer understands that everything belongs to
the same system.

When you finish a phase, run the test gate. If it doesn't pass, do not
start the next phase. The test gate is the proof that the northstar moved
closer, not just that code was written.

The goal is not to finish the build plan. The goal is to build Pangaea.
